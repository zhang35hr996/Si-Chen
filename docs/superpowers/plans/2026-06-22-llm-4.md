## Key findings

- `produceDialogueTurn` already returns the FULL `DialogueLine` (`turnResult.value.line`) with `expression`, `choices`, `meta`. `App.tsx` line 906 throws all of it away except `.text`. The data is already there тАФ #13 is purely a UI-wiring problem.
- `assembleDialogueRequest` already accepts `options.transcript` and threads it onto `request.transcript`. The multi-turn machinery exists; only the UI loop is missing.
- The claim gate's forbidden check (`isCoveredByForbiddenClaim`, claimGate.ts:140-152) is already **source-independent** тАФ fact key (predicate+subjectId+object) + polarity only, NO sourceRef check. So #14's "source-independence" acceptance criterion is satisfied by the existing gate once a forbidden claim flows through. The work is purely populating `assembled.forbidden`.
- `buildDialoguePolicyContext` (orchestrator.ts:265-289) reads `request.promptContext.forbiddenClaims` and threads it to the gate (only when non-empty). So once `assembleClaims.forbidden` is populated AND threaded into `promptContext.forbiddenClaims` (already wired at orchestrator.ts:131), the existing `forbidden_claim_test` fixture flows end-to-end.
- The `forbidden_claim_test` fixture already exists in `builders.ts:434-458`, but NO scenario row in `scenarios.jsonl` references it, and its docstring (builders.ts:426-433) describes a stale provider-hack approach that #14 invalidates.
- `produceDialogueLine` / `produceDialogueLineWithPolicy` are the **current, live** private function names in orchestrator.ts тАФ they are NOT stale and must NOT be removed for #15.
- Stale PR-number refs in source comments: `orchestrator.ts:234`, `gates.ts:7`, `reactionTypes.ts:11`, `state/types.ts:397`.

### Group A тАФ Issue #13: Generative DialogueLine + choices to UI

#### T1 тАФ Extend the reaction render contract to optionally carry a full DialogueLine

**Goal:** Let the UI carry a generated `DialogueLine` (expression/meta/choices) alongside the existing scripted `lines: string[]`, without breaking `DecreeReaction`.

**Files:**
- `src/ui/App.tsx` тАФ `reaction`/`reactionQueue` state shape (lines ~116, ~149) and `<ReactionScreen>` props block (~1333-1363)
- `src/ui/screens/ReactionScreen.tsx` тАФ accept an optional pre-built `DialogueLine`

**Approach:**
- Keep `DecreeReaction.lines: string[]` untouched. Add an **optional** field to the App-local `reaction` state object: `generatedLine?: DialogueLine`. When present, `ReactionScreen` renders it directly instead of calling `mockProvider`.
- `ReactionScreen`: add optional prop `generatedLine?: DialogueLine`. Branch in the `useEffect`: if `generatedLine` is supplied and `index === 0`, set it directly (skip `assembleDialogueRequest` + `produceDialogueTurn`). For scripted `lines`, keep the existing `mockProvider` path verbatim.

**Acceptance:**
- Scripted reactions render identically to before.
- When `generatedLine` is passed, `ReactionScreen` renders that exact expression/text without invoking `mockProvider`.

**Test approach:** Extend `tests/ui/dialogueGenerative.test.ts`: assert that a built `DialogueLine` round-trips its `expression`/`meta`/`choices` into the reaction state shape. Add a regression assertion that the scripted path still produces `lines: string[]`.

#### T3 тАФ Wire `converse()` to pass the full line and drive the next turn on choice click

**Goal:** Replace the lossy `playReactions([{ lines: [generatedLine.text] }, ...])` with a path that carries the full line and, on choice click, assembles + commits the next generative turn with an appended transcript.

**Files:**
- `src/ui/App.tsx` тАФ `converse()` (lines ~885-917) and `<ReactionScreen>` `onChoice` wiring (~1333-1363)

**Approach:**
- In `converse()` generative success branch: instead of `playReactions([{ speakerId: charId, lines: [generatedLine.text] }, ...])`, set the reaction state with `generatedLine` so the full line renders. Decree beats still enqueue as scripted `lines` after the generative line.
- Add an `onChoice` handler wired into `<ReactionScreen>`. On choice click:
  1. `const transcript = [...prevTranscript, { speaker: charId, text: currentLine.text }, { speaker: "player", text: chosenChoice.text }]`
  2. `expectedState = store.getState()` (re-snapshot AFTER previous CAS)
  3. `assembleDialogueRequest(db, expectedState, charId, location, { transcript })`
  4. `produceDialogueTurn(db, dialogueProvider, req, expectedState, logger)`
  5. `store.commitDialogueState(expectedState, turnResult.value.nextState)` тЖТ on success, `doAutosave()` and set reaction state to new `generatedLine`; carry transcript forward.
  6. On any failure: close the generative loop gracefully. **Do not** spend additional AP.
- Keep the transcript in App-local state so each choice appends.

**Acceptance:**
- Generative success shows the full line (correct portrait expression).
- Clicking a choice drives a CAS-guarded next turn with appended transcript, no extra AP.
- Fallback paths never black-screen.

**Test approach:** Extend `tests/ui/dialogueGenerative.test.ts`: "choice click drives turn 2 with appended transcript" тАФ assemble request with a 2-entry transcript, assert `request.transcript` carries both entries in order, run `produceDialogueTurn`, assert CAS sequencing matches the existing case-6 invariant.
