# LLM-3 Reaction Plan + Known Events + Claim Gates 接活

**日期:** 2026-06-22
**PR 目标:** 把 LLM-2 在 `DialoguePromptContext` 中留下的四个 stub（`reactionPlan` / `knownEvents` / `allowedClaims` / `forbiddenClaims`）填成真实值；在 `assembleDialogueRequest` 内一处统一装配；扩 eval 黄金集覆盖三条新链路。
**基准分支:** main（合并 LLM-2 `feat/llm-2-prompt-compiler` 后）
**worktree:** `.worktrees/llm-3`
**branch:** `feat/llm-3-reaction-claims`

---

## LLM-3 边界（来自总体 spec + LLM-2 非目标）

LLM-2 把编译器（`compilePromptPayload`）、prompt caching（`buildAnthropicToolRequest`）、eval runner（`runEvalScenarioWithProvider`）全部做完，并刻意把 `DialoguePromptContext` 的四个字段留作 stub。这四个 stub **当前唯一被写死的地方**是 `assembleDialogueRequest`（`src/engine/dialogue/orchestrator.ts`）：

```ts
const promptContext: DialoguePromptContext = {
  speakerDisplayName: resolveDisplayName(character, contextStanding, rank),
  rankDisplay,
  audience,
  relevantMemories: memCtx.activatedMemories.map(toPromptMemory),
  reactionPlan: undefined,   // ← LLM-3
  knownEvents: [],           // ← LLM-3
  allowedClaims: [],         // ← LLM-3
  forbiddenClaims: [],       // ← LLM-3
  choiceCandidates: [],
};
```

**LLM-3 范围（in scope）:**

1. **`knownEvents`**：把 `buildMemoryContext` 已经产出但被丢弃的 `knownEvents: CourtEvent[]` 经新 `toPromptEvent()` 转成 `PromptEvent[]` 填入 `promptContext.knownEvents`。
2. **`reactionPlan`**：把已存在的纯函数链 `deriveSubjectRelation` + `deriveDisposition` + 新 `eventToReactionContext()` + `planReaction()` 在 `assembleDialogueRequest` 内接活，产出真实 `ReactionPlan` 填入 `promptContext.reactionPlan`。无可规划事件时回退 neutral plan（非 undefined）。
3. **`allowedClaims` / `forbiddenClaims`**：调用已存在但未接线的 `assembleClaims(...)`，把 `allowed`/`forbidden` 填入 `promptContext`。
4. **`offeredContextIds` 扩展**：`knownEvents` 进 payload 后，`buildDialoguePolicyContext` 须把 event id 纳入 `offeredContextIds`，否则 provider 引用 event 来源会被 `unknown_source_context` 误拒。
5. **Eval 扩展**：新增黄金场景（覆盖 knownEvents 注入、claim allowed/forbidden gate、reactionPlan 落地）；新增 fixture builders；扩 `EvalScenario.expectations` 增 `mustKnowEventIds?`（断言 payload 含某 event）。

**LLM-3 不动（编译器/适配器已就绪）:**

- `compilePromptPayload`（已纯函数透传四字段，无改动）。
- `buildAnthropicToolRequest` / `WORLD_RULES_TEXT` / prompt caching（已就绪；规则 10/11 已引用 forbidden/allowed claims）。
- `runEvalScenarioWithProvider` 执行核心（只扩 `evaluateExpectations` 与 scenario/fixture，不改链路）。

## 非目标（显式推迟到 LLM-4+）

- **修复重试 / fallback**（gate 拒绝后回灌 provider 重生成）——LLM-4。
- **streaming**——LLM-4+。
- **让模型选 choice 文本**（choices 仍 authored；`choiceCandidates` 仍 `[]`，本 PR 不接）。
- **rumor / 错误信念 / certainty 投影**——claim gate 仍只读 `GroundTruthBeliefProjection`，谓词仍限 `resides_at|holds_rank|alive`。
- **多事件并发反应规划**：本 PR 只对「本回合最相关的单个 court event」规划反应；多事件合成留后续。
- **`self_feeling` / `relationship` 类 `claimNeed` 的结构化 claim**：`assembleClaims` 当前只对 `subject_event` 产 fact claim，本 PR 不扩。
- **关系/性格授定 schema 改造**：仍用 PR3 词表派生 + 诊断回退，不改 content 格式。
- **emotionalConditions 进 payload**：本 PR 不暴露情绪状态给 LLM。

## Global Constraints

1. **唯一装配点**：四个 stub 只在 `assembleDialogueRequest` 内填充。不在 `compilePromptPayload`、不在 provider、不在 UI 重算。`compilePromptPayload` 仍是纯透传。
2. **纯函数 / 确定性**：所有新派生函数（`toPromptEvent`、`eventToReactionContext`、`selectReactionEvent`、`buildReactionPlan`）同输入同输出；**禁 `Math.random()` / `Date`**；tie-break 用稳定排序。
3. **回退非空**：`reactionPlan` 在无可规划事件时回退 `NEUTRAL_REACTION_PLAN`（`primary: "remain_reserved"`），**不留 `undefined`**——这样 `assembleClaims` 的 `claimNeeds` 输入恒定义良好。`DialoguePromptContext.reactionPlan` 类型仍 `ReactionPlan | undefined`（兼容），但装配恒给值。
4. **来源单一不漂移（single-source invariant）**：`offeredContextIds`（gate）= provider 实际收到的来源 = `request.speakerContext.relevantMemories` 的 id ∪ `promptContext.knownEvents` 的 id。一旦 events 进 payload，gate 的 `offeredContextIds` 必须同步纳入 event id（否则合法 event 引用被误拒）。`buildDialoguePolicyContext` 据 `request.promptContext.knownEvents` 派生，不另算。
5. **gate 只读 `BeliefProjection`**：claim gate 不直读 ground truth 判「角色相信什么」。`assembleClaims` 的 forbidden 派生例外（读 `state.standing[*].residence`，因 belief 无 `currently_same_residence` 谓词）——此例外 PR5 已确立，本 PR 沿用。
6. **belief viewer = speakerId**：`assembleClaims` 的 belief 查询、`deriveSubjectRelation` 的 affection 来源、`planReaction` 的关系，viewer/owner 一律 speakerId。
7. **knownEvents 只含 speaker 可知事件**：`buildMemoryContext.knownEvents` 已经过 `canKnowEvent` 过滤，本 PR 直接消费，不重新过滤、不旁路 `canKnowEvent`。
8. **PromptEvent 不泄露内部字段**：`toPromptEvent` 去掉 `publicity` / `publicSalience` / `retention` / `tags`，只留 `id / type / occurredAt / participants / locationId? / facts`。`facts` 仅取 `CourtEvent.payload` 中的标量（string/number/boolean/null）。
9. **eval 不绕 compiler**：新场景与旧场景共用 `runEvalScenarioWithProvider`，prompt 仍来自 `compilePromptPayload → buildAnthropicToolRequest`。CI 用 fixture transport，不调真实 API。
10. **测试**：`npx vitest run` 全绿；`npx tsc --noEmit`（client+server）干净；`npx vite build` 成功（eval/工具不进 browser bundle）。每个 task 结束须全绿 + tsc clean。
11. **append-only / no-backcompat**：预发布不迁移旧档。
12. **称谓/礼制**遵循 dialogue-etiquette-rule 与 official-naming-rule。

## 关键接口（任务间约定）

### §1 `toPromptEvent`（T1 定义，T5 消费）

```ts
// src/engine/dialogue/promptPayload.ts —— 与既有 toPromptMemory 同文件
/** Strip engine-internal fields from a CourtEvent to produce a PromptEvent. */
export function toPromptEvent(e: CourtEvent): PromptEvent
```

- `facts` = 从 `e.payload` 过滤出标量：`Object.fromEntries(Object.entries(e.payload).filter(([, v]) => v === null || ["string","number","boolean"].includes(typeof v))) as Record<string, PromptFactValue>`。
- 透传 `id / type / occurredAt / participants`；`locationId` 仅当 `e.locationId !== undefined` 时带（exactOptionalPropertyTypes 干净）。
- 不含 `publicity / publicSalience / retention / tags`。

### §2 `eventToReactionContext`（T2 定义，T4 消费）

```ts
// src/engine/dialogue/eventReaction.ts（新建）
import type { CourtEvent } from "../state/types";
import type { EventReactionContext } from "./reactionTypes";

/**
 * 把一条 CourtEvent 映射为 planReaction 所需的 EventReactionContext。
 * 仅四类事件可规划反应；其余返回 undefined（本 PR 不规划）。
 * subjectId 取事件「主角」参与者。
 */
export function eventToReactionContext(
  event: CourtEvent,
  speakerId: string,
): EventReactionContext | undefined
```

映射规则（确定性、table 化）：

| `event.type` | `eventType` | `subjectId` 取自 participant role | `direction` |
|---|---|---|---|
| `heir_born` | `heir_born` | `role === "sovereign_parent"` 的参与者 ?? 首个非新生儿 | — |
| `heir_died` | `heir_died` | `role === "sovereign_parent"` 的参与者 ?? 首个非新生儿 | — |
| `rank_changed` | `rank_changed` | `role === "demoted"` 或 `"promoted"` 的参与者 ?? 首参与者 | 从 `payload.fromGrade`/`payload.toGrade` 比较或从 `role` 推 `demote`/`promote` |
| `residence_changed` | `residence_changed` | `role === "relocated"` 的参与者 ?? 首参与者 | — |
| 其余 | — | — | 返回 `undefined` |

- subjectId 解析不到参与者 → 返回 `undefined`（不造 id）。
- speaker 自指（`subjectId === speakerId`）仍返回 context（自评位分变动是合法反应）。

### §3 `selectReactionEvent`（T3 定义，T4 消费）

```ts
// src/engine/dialogue/eventReaction.ts（同文件）
/**
 * 从本回合 knownEvents 中选「最值得规划反应」的单个事件。
 * 过滤 eventToReactionContext 可映射者；稳定排序：occurredAt 降序
 * → publicSalience 降序 → id 升序。无可映射事件 → undefined。
 */
export function selectReactionEvent(
  events: readonly CourtEvent[],
  speakerId: string,
): { event: CourtEvent; reactionContext: EventReactionContext } | undefined
```

### §4 `buildReactionPlan`（T4 定义，T5 消费）

```ts
// src/engine/dialogue/reactionAssembler.ts（新建）
import type { ContentDB } from "../content/loader";
import type { GameState, CourtEvent } from "../state/types";
import type { DialogueAudienceContext } from "./audience";
import type { ReactionPlan } from "./reactionTypes";

export const NEUTRAL_REACTION_PLAN: ReactionPlan = {
  subjectIds: [],
  primary: "remain_reserved",
  intensity: 0,
  openness: 50,
  claimNeeds: [],
  rationaleCodes: ["llm3-no-event"],
};

/**
 * 装配真实 ReactionPlan：选事件 → 派生 relation/disposition → planReaction。
 * 无可规划事件 → NEUTRAL_REACTION_PLAN（恒定义良好，供 assembleClaims 消费）。
 * 纯函数确定性。
 */
export function buildReactionPlan(args: {
  db: ContentDB;
  state: GameState;
  speakerId: string;
  audience: DialogueAudienceContext;
  knownEvents: readonly CourtEvent[];
}): ReactionPlan
```

内部步骤：

1. `const picked = selectReactionEvent(args.knownEvents, args.speakerId)`；无 → `NEUTRAL_REACTION_PLAN`。
2. `subjectId = picked.reactionContext.subjectId`。
3. **relation**：取 speaker 对 subject 的 authored attitude（`db.characters[speakerId].stances?.find(s => s.charId === subjectId)?.attitude`）+ 运行时 `state.standing[speakerId]?.affection`（若 subject 是 player）；`favorThreat` 取 `state.standing[subjectId]?.favor`（无则 0）。`deriveSubjectRelation({ charId: subjectId, authoredAttitude, standingAffection, favorThreat }).relation`。
4. **disposition**：`deriveDisposition(db.characters[speakerId].profile.personalityTraits).disposition`。
5. **audience 适配**：`planReaction` 要 `AudienceContext`（reactionTypes），`DialogueAudienceContext` 已含 `targetRole / privacy / presentCharacterIds` — 直接结构子集传入：`{ targetRole: audience.targetRole, privacy: audience.privacy, presentCharacterIds: audience.presentCharacterIds }`。
6. `return planReaction({ relation, disposition, audience: adapted, event: picked.reactionContext })`。

## 已知实现期决策（须实现者核对）

- participant `role` 字符串（`demoted`/`promoted`/`sovereign_parent`/`relocated`/`newborn`）须对照真实 content/事件生产者实核（T2）；解析不到 → undefined，不造 id。
- `rank_changed` 的 `direction` 推断优先用 `payload.fromGrade`/`toGrade` 比较，缺则按 `role` 兜底（T2）。
- `forbiddenClaimSubjects` 首选不单列 finding，复用 `forbiddenTexts`；仅 `required_event_not_known` 新增（T7）。
- fixture 角色/位分 id 必须 `loadRealContent()` 实核，不硬编不存在 id（T4/T7）。
- knownEvents 数量受 `buildMemoryContext` topN（默认 5）约束；events 与 memories 共享 top-N 名额（PR4 rerank 行为），本 PR 不改。
