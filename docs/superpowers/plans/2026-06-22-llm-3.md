# LLM-3 Reaction Plan + Known Events + Claim Gates 接活

**日期:** 2026-06-22 (v2 修订)
**PR 目标:** 把 LLM-2 在 `DialoguePromptContext` 中留下的四个 stub（`reactionPlan` / `knownEvents` / `allowedClaims` / `forbiddenClaims`）填成真实值，并形成完整 policy 闭环：claim gate 强制执行 allowed/forbidden 列表；event 来源与具体 claim 绑定；writeback 分离 memory/event ID。
**基准分支:** main（合并 LLM-2 `feat/llm-2-prompt-compiler` 后）
**worktree:** `.worktrees/llm-3`
**branch:** `feat/llm-3-reaction-claims`

---

## LLM-3 边界

LLM-2 把编译器（`compilePromptPayload`）、prompt caching（`buildAnthropicToolRequest`）、eval runner（`runEvalScenarioWithProvider`）全部做完，并刻意把 `DialoguePromptContext` 的四个字段留作 stub。这四个 stub **当前唯一被写死的地方**是 `assembleDialogueRequest`（`src/engine/dialogue/orchestrator.ts`）：

```ts
const promptContext: DialoguePromptContext = {
  ...
  reactionPlan: undefined,   // ← LLM-3
  knownEvents: [],           // ← LLM-3
  allowedClaims: [],         // ← LLM-3
  forbiddenClaims: [],       // ← LLM-3
  choiceCandidates: [],
};
```

**LLM-3 范围（in scope）:**

1. **`knownEvents`**：把 `buildMemoryContext` 产出的 `knownEvents: CourtEvent[]` 经 `toPromptEvent()` 转成 `PromptEvent[]`，填入 `promptContext.knownEvents`。
2. **`reactionPlan`**：`eventToReactionContext → selectReactionEvent → buildReactionPlan`——有合格事件才生成真实 plan；无则 `undefined`（不注入 neutral plan）。
3. **`allowedClaims` / `forbiddenClaims`**：升级 `assembleClaims` 输出为 `AuthorizedClaim[]`（claim + 授权 source ID），填入 prompt context；claim gate 强制执行 `claim_not_allowed` / `claim_explicitly_forbidden` / `source_not_authorized`。
4. **offeredContextIds 分离**：policy 中 `offeredMemoryIds` / `offeredEventIds` 拆开；claim gate 查 `offeredContextIds`（并集）；writeback 只查 `offeredMemoryIds`，event ID 不进 `MemoryMentionRecord`。
5. **Eval 扩展**：新增黄金场景覆盖 knownEvents / claim gate / reaction；新增 `mustKnowEventIds` expectation；新增 `required_event_not_known` finding code。

**LLM-3 不动（已就绪）:**
- `compilePromptPayload`（纯透传，无改动）
- `buildAnthropicToolRequest` / `WORLD_RULES_TEXT` / prompt caching
- `runEvalScenarioWithProvider` 执行核心

## 非目标（显式推迟到 LLM-4+）

- 修复重试 / fallback（gate 拒绝后重生成）——LLM-4
- streaming——LLM-4+
- 让模型选 choice 文本（`choiceCandidates` 仍 `[]`）
- rumor / certainty 投影替换 `GroundTruthBeliefProjection`
- 多事件并发反应规划
- `self_feeling` / `relationship` 类 claimNeed 扩展
- `emotionalConditions` 进 payload

## Global Constraints

1. **唯一装配点**：四个 stub 只在 `assembleDialogueRequest` 内填充。`compilePromptPayload` 仍是纯透传。
2. **纯函数 / 确定性**：所有新派生函数禁 `Math.random()` / `Date`；tie-break 用稳定排序。
3. **reactionPlan 无合格事件 → `undefined`**：不注入 `NEUTRAL_REACTION_PLAN` 到 prompt。`assembleClaims` 可接受 `reactionPlan?: ReactionPlan`（undefined 时 `claimNeeds = []`）。
4. **来源双轨不漂移**：`offeredContextIds = offeredMemoryIds ∪ offeredEventIds`；writeback 只用 `offeredMemoryIds`。event ID 不得出现在 `MemoryMentionRecord.memoryId`。
5. **AuthorizedClaim 双重绑定**：gate 检查（a）claim 在授权集合；（b）proposed source 是该 claim 的授权 source 之一；（c）source 已发给模型（在 `offeredContextIds`）。
6. **事件 participant role 与真实生产者对齐**（见 §2）：不能 fallback 到"首参与者"。
7. **belief viewer = speakerId**：所有 belief / affection / stance 查询 owner = speakerId。
8. **knownEvents 只含 speaker 可知事件**：直接消费 `buildMemoryContext.knownEvents`，不旁路 `canKnowEvent`。
9. **eval 不绕 compiler**：新场景共用 `runEvalScenarioWithProvider`，CI 用 fixture transport。
10. **测试**：每个 task 结束须 `npx vitest run` 全绿 + `npx tsc --noEmit` clean。
11. **append-only / no-backcompat**：预发布不迁移旧档。
12. **称谓/礼制**遵循 dialogue-etiquette-rule 与 official-naming-rule。

## 关键接口（任务间约定）

### §1 `toPromptEvent`（T1）

```ts
// src/engine/dialogue/promptPayload.ts（紧邻 toPromptMemory）
export function toPromptEvent(e: CourtEvent): PromptEvent
```

- `facts`：过滤 `e.payload` 标量，额外排除 `NaN` / `Infinity`：
  ```ts
  (v === null)
  || (typeof v === "string")
  || (typeof v === "boolean")
  || (typeof v === "number" && Number.isFinite(v))
  ```
- 透传 `id / type / occurredAt / participants`；`locationId` 条件透传。
- **不含** `publicity / publicSalience / retention / tags`。

### §2 `eventToReactionContext`（T2）

```ts
// src/engine/dialogue/eventReaction.ts（新建）
export function eventToReactionContext(
  event: CourtEvent,
  speakerId: string,
): EventReactionContext | undefined
```

**真实 participant role 映射**（来自实际事件生产者契约）：

| `event.type` | `subjectId` 取法 | `direction` 取法 |
|---|---|---|
| `rank_changed` | `participant("subject")` | `payload.direction` |
| `residence_changed` | `participant("mover")` | — |
| `heir_born` | `participant("adoptive_father") ?? participant("birth_father")` | — |
| `heir_died` | `participant("adoptive_father") ?? participant("birth_father")` | — |
| 其余（`punished`/`rewarded`/`conflict`/`promise`/`secret_discovered`） | — | 返回 `undefined` |

**规则：**
- 不能 fallback 到"首参与者"（避免把 `newborn`/`deceased` 当关系对象）。
- `payload.direction` 字段名：`from` / `to` / `direction`（**不是** `fromGrade`/`toGrade`）。
- subjectId 解析不到 → 返回 `undefined`。
- `speakerId === subjectId` 仍合法（自评位分变动）。

helper:
```ts
function participant(role: string, event: CourtEvent): string | undefined {
  return event.participants.find(p => p.role === role)?.charId;
}
```

### §3 `selectReactionEvent`（T3）

```ts
// src/engine/dialogue/eventReaction.ts
export function selectReactionEvent(opts: {
  events: readonly CourtEvent[];
  speakerId: string;
  targetId: string;
  sceneDirective?: string;
  now: GameTime;
  presentCharacterIds: readonly string[];
}): { event: CourtEvent; reactionContext: EventReactionContext } | undefined
```

**Eligibility 过滤（全部满足才入选）：**

1. `eventToReactionContext(event, speakerId)` 可映射（非 undefined）。
2. 事件与本轮对话相关——满足其一：
   - `event.participants` 含 `opts.targetId`；
   - `event.participants` 含 `opts.presentCharacterIds` 中的任意人；
   - 事件发生于本局（无需时间比较，有 subject 可知即可）。
3. 存在显式 `sceneDirective` 时，**跳过**该选择（`return undefined`）——由 directive 主导本轮行为目标，不让事件覆盖。

**排序**（多个合格事件）：`occurredAt` 降序 → `publicSalience` 降序 → `id` 升序。

**无合格事件 → `undefined`。**

### §7 Claim gate 新增违规码（T6）

```ts
// src/engine/dialogue/gates.ts（现有 ClaimViolationCode 追加）
type ClaimViolationCode =
  | ...existing...
  | "claim_not_allowed"         // fact claim 不在 allowedClaims 集合中
  | "claim_explicitly_forbidden" // fact claim 命中 forbiddenClaims 集合
  | "source_not_authorized";    // proposed source 不是该 claim 的授权 source

// gate 使用的语义 key（不依赖 LLM 生成的 claim ID）
function claimSemanticKey(claim: DialogueClaim): string {
  return JSON.stringify([
    claim.predicate,
    claim.subjectId,
    claim.object ?? null,
    claim.modality,
  ]);
}
```

Gate 检查顺序（在现有 `unknown_source_context` 之后插入）：
1. 若 policy 有 `forbiddenClaims`：`claimSemanticKey(proposed)` 命中 → `claim_explicitly_forbidden`。
2. 若 policy 有 `allowedClaims`（非空）：key 不在集合 → `claim_not_allowed`。
3. 若 proposed 的 source 不在该 claim 的 `sourceContextIds`（交 `offeredContextIds`）→ `source_not_authorized`。

- [ ] Claim gate 执行 `claim_not_allowed` / `claim_explicitly_forbidden` / `source_not_authorized`；使用 semantic key
- [ ] `offeredContextIds = offeredMemoryIds ∪ offeredEventIds`；writeback 只查 `offeredMemoryIds`；event ID 不进 `MemoryMentionRecord`
- [ ] 四字段只在 `assembleDialogueRequest` 装配；`compilePromptPayload` 零改动
